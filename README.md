# 特休假管理系統 — 部署說明

系統跑在 **Google Apps Script**（免費、免主機），資料庫就是您的 Google Sheet。以下步驟約 10-15 分鐘。

## 一、建立試算表與程式

1. 開一個**空白試算表**，取個名字（例如「特休假管理系統」）。不用自己建立分頁或欄位，程式會自動建立。
2. 上方選單「擴充功能 → Apps Script」，會開啟一個新分頁的 Apps Script 編輯器。
3. 編輯器裡預設會有一個 `Code.gs`，把裡面的內容**全部刪除**，貼上本次提供的 **Code.gs** 內容。
4. 點編輯器左側「檔案」旁的 **+ → HTML**，檔名輸入 `Index`（大寫 I，不用加 `.html`），貼上本次提供的 **Index.html** 內容。
5. 存檔（Ctrl/Cmd + S）。

## 二、初始化資料表

1. 回到 Google Sheet 分頁，重新整理網頁。
2. 上方選單會多一個「**特休系統**」選單，點選「初始化 / 檢查資料表結構」（第一次執行會要求 Google 帳號授權，屬正常流程）。
3. 執行後會自動建立 3 個分頁：
   - **員工資料**（人事基本資料）
   - **請假紀錄**
   - **系統設定**（HR / 主管白名單）
   - **LINE通知設定**（主管 LINE Notify Token）
   - **新進員工待審核**

## 三、設定 HR / 主管白名單

到「系統設定」分頁，把可以使用 HR 後台（新增員工、審核假單）的 Email，一列填一個（從第 2 列開始）。

## 四、部署成公開網址

1. Apps Script 編輯器右上角「**部署 → 新增部署作業**」，類型選「網頁應用程式」。
2. 執行身分：**我**；誰可以存取：**任何人**（若只想公司內部使用且是 Google Workspace 網域帳號，可改選「XX 機構中的任何人」）。
3. 部署後會得到公開網址，可直接分享給員工使用。
4. **之後修改程式碼**：部署 → 管理部署作業 → 編輯（鉛筆圖示）→ 版本選「新版本」→ 部署，否則員工看到的還是舊版。

## 「員工資料」分頁欄位與唯一識別

**這一版改用「身份證字號」作為員工的唯一識別**，不再有系統自動產生的員工編號。新增員工時，HR 後台會請你填以下欄位：

店點、員工類型、職位、姓名、暱稱、到職日、**身份證字號**、出生年月日、性別、戶籍地址、通訊地址、家用電話、個人手機、學歷、籍貫、投保日期、薪資、緊急聯絡人、關係、緊急聯絡人手機、備注、IG、LINE ID、離職日期、**銀行帳號**

- **身份證字號**：必填、需唯一，同時也是**員工登入查詢時使用的憑證**（姓名 + 身份證字號），也是串連「員工資料」與「請假紀錄」兩個分頁的識別依據。新增後請勿更改，若真的打錯需要修改，直接到「員工資料」分頁手動改該筆儲存格即可（同時注意「請假紀錄」分頁裡這位員工過去的申請紀錄不會自動連動更新，需要一併手動修改對應欄位）。
- **銀行帳號**：純粹是薪資轉帳用的資料欄位，非必填，**不會**用於登入或查詢，只有 HR 後台看得到。
- **離職日期**：若填寫，員工總覽會自動顯示「離職」狀態；留空則顯示「在職」。

> **提醒**：身份證字號是敏感個資，這個設計會讓它同時出現在「請假紀錄」分頁（每筆假單都會記錄）與 HR 後台的待審核假單列表上。如果之後想降低曝光範圍，可以考慮改回系統自動編號、僅在「員工資料」分頁保留身份證字號當作單純的個資欄位——這部分我可以隨時幫你改回來。

## 新增功能：新進員工資料填寫（公開表單）

前端現在多了第三個入口「**新進員工資料填寫**」，任何人都能打開網址填寫，欄位跟 HR 後台「新增員工」完全相同。填寫送出後**不會直接變成正式員工資料**，而是先進到「新進員工待審核」，HR 後台需要按「核准建立」才會正式寫入「員工資料」分頁；按「忽略」則會標記為已處理但不建立。這樣設計是為了避免任何人透過公開網址直接寫入正式員工資料，同時讓新人自己填寫繁瑣的個人資料，減少 HR 手動輸入的工作量。

所有提交紀錄（不論核准或忽略）都會保留在「新進員工待審核」分頁，方便日後查核。

## 新增功能：員工詳細資料 / 編輯（含薪資調整）

「員工總覽」的每一列現在都可以**點擊**，會跳轉到「員工詳細資料」頁面，顯示這位員工完整的欄位（含目前特休狀態），並且可以直接編輯任何欄位後按「儲存變更」寫回「員工資料」分頁。

這解決了兩件事：
1. **調薪**：薪資會逐年調整，現在不用再手動去試算表改儲存格，直接在網頁上點選員工 → 改「薪資」欄位 → 儲存即可，往後的特休折抵金額試算會自動用新薪資計算。
2. 其他欄位（店點異動、職位晉升、聯絡方式更新、離職日期登記等）也都可以用同一個頁面直接改，不用開 Google Sheet。

> 身份證字號在這個頁面也是可編輯的（例如打錯字需要更正），但因為它是串連「請假紀錄」的識別鍵，如果真的更改了，過去的請假紀錄不會自動連動更新，需要另外手動處理（同前面提過的注意事項）。

## 關於「入職日在系統上線前」的員工：歷史特休折抵計算修正

如果員工的到職日早於「HR 把他建檔進系統的日期」（例如到職日是 2023/11/1，但 2026 年才把資料建進這個系統），代表這位員工過去已經走過的特休週期（例如到職第 1 年、第 2 年）**在系統裡完全沒有請假紀錄**——不是因為他沒休假，而是那些假是在系統上線前就休掉、或早就跟公司結清了。

如果照系統原本的邏輯無腦計算，會誤以為「這些週期完全沒使用」，進而在「歷年特休紀錄」顯示一筆錯誤的到期未休天數與折抵金額，這是不準確的。

**這次已經修正**：系統現在會記住每位員工是「哪一天被建檔進系統」（用新增當下的建立時間），**只有在建檔日之後才結束的特休週期，才會被列入「歷年特休紀錄」的到期折抵計算**；建檔之前就已經結束的週期，會直接略過、不顯示在歷史紀錄裡，避免出現不準確的金額。至於「目前所在的特休週期」與「未來的週期」，仍然會正常依照到職日的週年制持續往下算，年資本來就是連續累計的，不會因為系統上線時間而重新歸零或中斷。

換句話說：像你提到 2023/11/1 到職、年資持續累計的這批員工，只要把他們的**到職日**如實填入系統（不用去換算或调整），目前與未来的特休天數就會自動正確反映其實際年資；系统上線前已经过去、無資料可查证的週期，則不会显示错误的历史折抵金额。

## ⚠️ 既有試算表需要手動補欄位（重要）

這次更新在既有的資料結構上多了 2 個欄位。因為 Apps Script 的「初始化」只會在分頁全新空白時建立表頭，**不會自動幫你改已經有資料的分頁**，所以如果你已經在用這個系統、已經有真實資料，請手動到 Google Sheet 做以下兩件事：

1. **「員工資料」分頁**：在「銀行帳號」欄位後面（「建立時間」欄位前面）**插入一欄**，欄名填「Email」
2. **「請假紀錄」分頁**：在「拒絕原因」欄位後面（最後一欄）**新增一欄**，欄名填「假別」

兩個都只是「加一欄」，不會動到既有資料，插入/新增後直接重新貼上這次的 Code.gs 和 Index.html 即可正常運作。

如果你是全新安裝、還沒有任何資料，直接照最上面「初始化」的步驟做就會自動建好，不用手動處理這一段。

## 新增功能：多假別請假

系統現在支援 6 種假別：**特休、事假、病假、婚假、喪假、產假**。員工在「請假申請」表單選擇假別後，畫面會即時顯示該假別對應的天數資訊。

各假別的計算規則不同：

| 假別 | 規則 | 系統行為 |
|---|---|---|
| 特休 | 到職滿 6 個月起，依年資級距遞增（3/7/10/14/15…天），週年制、到期不遞延 | 自動檢查剩餘天數，超過無法送出 |
| 事假 | 每年固定 14 天，到職當天即可用，週年制 | 自動檢查剩餘天數，超過無法送出 |
| 病假 | 每年固定 30 天，到職當天即可用，週年制 | 自動檢查剩餘天數，超過無法送出 |
| 婚假 | 預設 8 天（僅供參考） | **不自動限制**，天數合理性由主管審核判斷 |
| 喪假 | 預設 8 天（僅供參考，實際依親等不同） | **不自動限制**，由主管審核判斷 |
| 產假 | 預設 56 天（僅供參考） | **不自動限制**，由主管審核判斷 |

> 婚假/喪假/產假之所以不做自動額度檢查，是因為這幾種假別是「一次性事件假」而非年度重複配額，且喪假天數依親等不同、公司也可能給予優於法定的天數，交由主管人工判斷比系統寫死規則更合理。如果之後想讓系統也自動核算，跟我說實際規則（例如喪假依親等分級距），可以再調整。這些天數都定義在 `Code.gs` 最上面的 `LEAVE_TYPES` 常數裡，之後要調整天數直接改那邊的數字即可。

「請假紀錄」分頁多了一欄「**假別**」（放在最後一欄，方便既有資料直接補欄，不會打亂原本欄位）。「我的請假紀錄」與 HR 後台「待審核假單」也都會顯示假別。

## 新增功能：Email 通知

- **員工送出請假申請時**：系統會自動寄信給「系統設定」分頁裡所有的 HR/主管白名單 Email，內容包含姓名、假別、期間、天數、事由。
- **HR 核准或拒絕假單時**：系統會自動寄信通知該員工審核結果（核准／拒絕，拒絕的話會附上原因）。

**這個功能需要員工資料有 Email**，因此「員工資料」欄位最後新增了一欄「**Email**」（放在「銀行帳號」後面、「建立時間」前面）。新增員工、新進員工填寫表單、員工詳細資料編輯，都新增了 Email 輸入欄位。

**注意事項**：
- Email 是**非必填**欄位；如果某位員工沒有填 Email，核准/拒絕時系統就不會寄信給他（但仍會正常執行核准/拒絕，不影響功能）。
- 寄信是用你 Google 帳號本身的 Gmail 額度寄出（一般 Gmail 帳號每天約 100 封，Workspace 帳號更高），內部員工人數不多的話完全夠用。
- **第一次觸發寄信功能時**，Apps Script 可能會要求重新授權（因為多用到了寄信的權限），照畫面指示同意即可，這是正常流程。
- 如果之後想額外用 LINE 通知：原本很多人用的「LINE Notify」服務已經在 2025 年 3 月正式終止，現在要改用「LINE Messaging API」，需要申請 LINE 官方帳號、設定 webhook，且員工要先加公司官方帳號好友才能收到訊息，門檻明顯更高，算是另一個獨立專案，之後有需要可以再討論。

## 新增功能：主畫面 4 宮格

前端首頁改成 4 個方塊：

- **員工查詢**：輸入姓名+身份證字號後，直接看到「特休查詢」頁籤（特休狀態、歷年紀錄、其他假別總覽）
- **請假申請**：輸入姓名+身份證字號後，直接看到「請假申請」頁籤（選假別、填日期送出、我的請假紀錄）
- **新進員工資料填寫**：跟之前一樣，公開表單，送出後待 HR 審核建立
- **HR‧主管後台**：跟之前一樣，白名單 Email 登入

「員工查詢」跟「請假申請」其實是同一個登入身份（同一組姓名+身份證字號），差別只在於**點進去後預設顯示哪一個頁籤**；登入後畫面上方也有頁籤可以自己切換，不用重新查詢。

## 新增功能：手機推播通知（OneSignal，僅限 GitHub Pages 版）

除了 Email 通知，現在同一時間點也會發送**手機推播通知**（像原生 App 的通知那樣）：
- 員工送出請假申請 → 推播通知所有已登入過 HR 後台的裝置
- HR 核准/拒絕 → 推播通知該員工（登入過「員工查詢」或「請假申請」的裝置）

**這個功能只在 GitHub Pages 的 PWA 版本生效**，Apps Script 內嵌的網頁版不支援（原因跟之前說明的「無法做成真正 App」一樣：Apps Script 的網頁是跑在 Google 網域的沙盒 iframe 裡，沒辦法註冊真正的推播訂閱）。

### 設定步驟

1. **申請 OneSignal 帳號**：前往 https://onesignal.com ，免費註冊
2. **建立一個 App**：選擇平台「Web」，網站網址填你的 GitHub Pages 網址（例如 `https://yourname.github.io/leave-app/`），照畫面精靈設定完成
3. **取得金鑰**：進到這個 App 的「Settings → Keys & IDs」，複製 **App ID** 跟 **REST API Key**
4. **貼到 Code.gs**：找到這幾行，換成你自己的值：
   ```js
   const ONESIGNAL_APP_ID = 'PASTE_YOUR_ONESIGNAL_APP_ID_HERE';
   const ONESIGNAL_REST_API_KEY = 'PASTE_YOUR_ONESIGNAL_REST_API_KEY_HERE';
   ```
   貼完部署新版本。
5. **貼到 app/index.html**：找到這一行，換成你的 App ID：
   ```js
   await OneSignal.init({ appId: "PASTE_YOUR_ONESIGNAL_APP_ID_HERE" });
   ```
   存檔上傳到 GitHub。
6. 打開 GitHub Pages 網址，用「員工查詢」或「HR‧主管後台」登入，瀏覽器應該會跳出「允許通知？」的請求，同意即可完成訂閱測試。

### 重要提醒

- **iPhone 使用者**：iOS 的網頁推播只有「已加入主畫面」的 PWA 才能收到通知，且需要 iOS 16.4 以上版本；如果只是用 Safari 開網頁沒有加到主畫面，是收不到推播的，請提醒同事務必先「加入主畫面」再允許通知。
- 如果 `ONESIGNAL_APP_ID` 還是預設的提示文字（沒有換成真的值），系統會自動略過推播、不會報錯，Email 通知仍會正常運作，不影響其他功能。
- 這是我依照 OneSignal 目前公開文件實作的版本；第三方服務的 API 偶爾會調整，如果設定後推播沒有跳出來，建議先到 OneSignal 後台的「Web Push 設定精靈」對照一下它產生的程式碼跟這裡的是否一致（尤其是 `app/index.html` 裡 OneSignal SDK 那段），有落差的話用它精靈產生的版本為準。
- HR 的推播是用「標籤」鎖定所有登入過後台的裝置，員工的推播是用「身份證字號」當作專屬 ID 鎖定，兩者互不影響。

## 新增功能：LINE 通知

除了 Email 跟 OneSignal 推播，現在也支援 **LINE 通知**——員工送出請假申請時通知 HR，審核完成時通知員工，訊息會直接發送到大家的 LINE。這個功能**兩個前端都能用**（Apps Script 內嵌版跟 GitHub Pages PWA版皆可），跟 Email 是一樣的道理。

### 為什麼綁定方式是「驗證碼」而不是直接輸入身份證字號？

因為 Apps Script 沒辦法讀取 LINE 傳來的簽章表頭（`X-Line-Signature`），沒辦法驗證一個 Webhook 請求「真的是從 LINE 官方伺服器來的」。如果綁定方式是「在 LINE 對話框輸入自己的身份證字號」，任何知道你身份證字號的人都可以冒充輸入，把你的通知綁到他自己的 LINE 帳號上——這是不安全的。

所以改成：**先登入系統、點「取得綁定驗證碼」**，系統會給一組限時 10 分鐘的 6 位數驗證碼，**在 LINE 對話框輸入這組驗證碼**才會完成綁定。因為要先通過系統登入（員工要輸入姓名+身份證字號、HR 要輸入白名單 Email）才能拿到驗證碼，安全性比直接用身份證字號高出不少。

### 設定步驟

1. **申請 LINE 官方帳號**：
   - 前往 https://developers.line.biz/zh-hant/ ，用你的 LINE 帳號登入
   - 建立一個 Provider（服務提供者，可以想成是「公司」這個概念）
   - 在 Provider 底下「Create a Messaging API channel」→ 「Create a LINE Official Account」，填寫帳號名稱等資訊完成建立
2. **啟用 Messaging API**：進入剛建立的官方帳號後台 → 設定 → Messaging API → 啟用（選擇剛剛的 Provider）
3. **關閉自動回應、開啟 Webhook**：
   - 在官方帳號管理後台「回應設定」，把「Webhook」打開、「自動回應訊息」關閉（不然員工加好友時 LINE 會自動回罐頭訊息，干擾我們自己的回覆邏輯）
4. **取得金鑰**：回到 LINE Developers Console 的 Messaging API 頁籤，複製 **Channel access token（long-lived）**
5. **設定 Webhook URL**：在同一個頁面找到 Webhook URL 欄位，貼上你的 Apps Script 部署網址（結尾 `/exec` 那組），存檔後點旁邊的「Verify」測試應該要成功
6. **取得官方帳號 Basic ID**：在 LINE Official Account Manager 後台可以找到你的帳號 ID（`@`開頭那組，例如 `@123abcde`）
7. **貼到 Code.gs**：找到這兩行，換成你自己的值：
   ```js
   const LINE_CHANNEL_ACCESS_TOKEN = 'PASTE_YOUR_LINE_CHANNEL_ACCESS_TOKEN_HERE';
   const LINE_OA_ID = 'PASTE_YOUR_LINE_OA_BASIC_ID_HERE'; // 例如 @123abcde
   ```
   貼完部署新版本
8. 到 Google Sheet 執行一次「特休系統 → 初始化」，會自動建立一個新分頁「**LINE綁定**」（存放綁定紀錄用，不用手動編輯）

### 使用方式

- 員工：登入「員工查詢」後，在「特休查詢」頁籤會看到「LINE 通知綁定」卡片，點「取得綁定驗證碼」，照畫面指示加官方帳號好友、輸入驗證碼即可
- HR：登入後台後，最上面就有「LINE 通知綁定」卡片，操作方式一樣

### 注意事項

- LINE 官方帳號免費方案每月有 200-500 則免費推播額度（依方案而定），一般公司內部使用應該綽綽有餘
- 如果 `LINE_CHANNEL_ACCESS_TOKEN` 還是預設的提示文字，系統會自動略過 LINE 通知、不會報錯，Email 和 OneSignal 通知仍正常運作
- 同一個人如果重新點「取得驗證碼」並完成綁定，會自動取代舊的綁定紀錄（不會產生重複）
- 這是我依照 LINE 目前公開文件實作的版本，如果 Messaging API 介面之後有調整，可能需要對照 LINE Developers Console 當下的實際畫面微調設定步驟

## 新增功能：特休數據同步到試算表（含每日自動排程）

「員工資料」分頁最後面現在會自動多出 4 欄：**特休天數(本期)、已用天數(本期)、剩餘天數(本期)、特休更新時間**。這幾欄是「快照」，不是即時公式——只有在以下情況才會更新：

- 員工自己查詢的時候（該員工那一列會更新）
- HR 在「員工總覽」點「**同步全部員工的特休數據到試算表**」按鈕（所有人一次更新）
- 設定好每日自動排程之後，系統每天會自動幫全部員工更新一次

### 設定每日自動排程

1. 打開 Apps Script 編輯器
2. 上方函式選單選擇 **`createDailySyncTrigger`**
3. 點「執行」（▶ 圖示）
4. 第一次執行會要求授權，照畫面同意即可
5. 跳出「已建立每日自動同步排程」的提示，代表設定成功——之後系統會**每天凌晨 2 點左右**自動把所有員工的特休數據刷新寫入試算表，不用再手動點按鈕

如果之後想取消或確認排程還在不在：Apps Script 編輯器左側選單「**觸發條件**」（時鐘圖示），可以看到 `dailySyncAllLeaveStats` 這個排程，能在那裡刪除或修改執行時間。

## 效能優化與防止連續點擊送出多筆

這次也順便處理了兩個問題：

1. **查詢/列表變慢的問題**：原本「員工總覽」在列出所有人時，每一位員工都會各自重新讀一次「請假紀錄」整份資料，人數一多就會明顯變慢。這次改成整個列表只讀一次共用資料，人數越多、改善越明顯。
2. **連續點擊送出多筆重複資料**：請假申請、新增員工、新進員工填寫、儲存員工資料、審核假單/新進員工這幾個按鈕，現在點下去後**會立刻鎖定並顯示「處理中…」**，直到系統回應完成才會恢復，避免手滑連點造成同一筆資料被送出好幾次。

## 新增功能：HR 白名單對應姓名

「系統設定」分頁現在是 **2 欄**：A 欄是 Email（跟原本一樣），**B 欄新增「姓名」**。填了姓名之後：

- HR 登入後台，畫面最上面會顯示「您好，OOO」
- 之後這位主管核准/拒絕請假，或審核新進員工資料，「請假紀錄」「新進員工待審核」分頁的「審核人／處理人」欄位會記錄**姓名**，而不是一長串 Email，方便日後查閱是誰審核的
- 如果某個 Email 沒有填姓名，系統會自動改顯示 Email，不會出錯

**既有的「系統設定」分頁記得手動在 B 欄補上對應的姓名**，一列一個，跟 A 欄的 Email 對應。

## 新增功能：排休申請（四週變形工時，A/B 區塊 + 25 號開放下月）

這是一個跟「請假」完全獨立的新模組——排休是四週變形工時下的正常輪休安排，不是特休/事假那種請假事件，所以資料表、審核流程都是分開的。

### 為什麼有 A、B 兩個區塊？

因為四週變形工時的週期起始日不是每個月 1 號，但班表還是照日曆月份（1號~月底）呈現，所以每個月會被切成 **A、B 兩個區塊**，分別對應到不同的變形工時週期，**各自有各自獨立的可休天數**（不是整個月共用一個數字）。

### 運作邏輯

1. **每年底先設定好隔年每月的 A/B 區塊與可休天數**：到 Google Sheet「**排休設定**」分頁，填「年月(YYYY-MM)、店點、區塊(A或B)、起始日、結束日、可休天數」。同一個年月+店點會有 **2 列**（A 一列、B 一列），例如：

   | 年月(YYYY-MM) | 店點 | 區塊 | 起始日 | 結束日 | 可休天數 |
   |---|---|---|---|---|---|
   | 2027-01 | 皮卡一店 | A | 2027-01-01 | 2027-01-15 | 4 |
   | 2027-01 | 皮卡一店 | B | 2027-01-16 | 2027-01-31 | 4 |

   這些起訖日跟天數系統都不會自動算，需要 HR 每年底先算好填進去。

2. **每月 25 號才開放申請下個月**：員工沒辦法無限提前申請，系統會檢查「今天」是不是已經過了「目標月份的上個月 25 號」，沒過就不能送出申請，會顯示開放日期。這個開放日「25 號」寫在 `Code.gs` 的 `SCHEDULE_OPEN_DAY` 常數，如果之後要改成別的日期，改這個數字就好。

3. **員工自己提交想要的休假日**：員工登入「員工查詢」→「排休申請」頁籤，選年月後會看到 **A、B 兩個區塊**各自的可休天數/已申請/剩餘，選日期提交（系統會自動判斷這天屬於哪個區塊，用該區塊的配額檢查，不是用整月合計）。

4. **系統標記衝突，但不擋下申請**：如果同一天已經有其他「技術師」申請排休，系統會提醒該名員工「這天已有其他技術師申請」，但**仍會送出**，因為最終要由主管協調決定怎麼安排，不是系統自動核准/拒絕。

5. **主管在後台協調**：HR 後台「排休協調」分頁，選店點+年月，可以看到 A/B 兩區塊的配額使用狀況，以及整個月所有人的申請（含所屬區塊），衝突的日期會用淺色底標示「⚠衝突」，主管逐筆按「確認」「需調整」或「拒絕」。

6. **店點固定公休**：對眼店每週一公休，系統會自動識別這天、提醒員工不需要申請（這個規則寫在 `Code.gs` 的 `STORE_CLOSURE_WEEKDAY` 常數裡，如果之後公休規則變動，或想加其他店的固定公休日，跟我說一聲就能調整）。

### 「同一天不可兩位技術師同時休假」的判斷方式

系統只針對「職位＝技術師」的員工做這個檢查（比對「員工資料」分頁的職位欄位）。如果職位打錯字或用了其他名稱（例如「資深技術師」），系統會判斷成不同職位、不會抓到衝突，請確保職位欄位統一填寫「技術師」三個字。

### 通知

員工送出排休申請時，會透過 Email／OneSignal／LINE（看你設定了哪些）通知 HR，跟請假申請用同一套通知管道。審核結果目前**不會**主動通知員工（員工需要自己回系統「排休申請」頁籤查看狀態），如果也想在確認/拒絕時通知員工，跟我說一聲可以加上去。

### 新增的分頁

- **排休設定**：HR 手動填寫的 A/B 區塊起訖日與可休天數表
- **排休申請**：員工提交的排休申請紀錄（申請編號、身份證字號、姓名、店點、職位、日期、狀態、申請時間、主管備注、處理人、處理時間）

## 系統操作方式

- **員工**：開啟網址 → 選「員工查詢」或「請假申請」→ 輸入姓名 + 身份證字號 → 查詢各假別天數、送出請假申請、查看歷史紀錄。畫面上只會顯示姓名、店點、職位等基本資訊與假別數據，不會把身份證字號、地址、緊急聯絡人等其他敏感欄位顯示回畫面上。
- **HR / 主管**：開啟網址 → 「HR‧主管後台」→ 白名單 Email 登入 → 新增員工（含完整人事欄位與 Email）、查看員工總覽、審核待審核假單（審核後自動寄信通知員工）。若要看完整人事資料（地址、緊急聯絡人等），直接開啟「員工資料」試算表分頁即可，那裡有完整內容。

## 系統邏輯與重要假設（請確認是否符合貴公司規定）

1. **特休天數**：依勞基法第 38 條標準級距計算，採**到職日週年制**，到期後不遞延。若貴公司規定不同，需調整 `Code.gs` 裡的 `buildPeriods_` 函式。
2. **到期折抵金額**：以「薪資 ÷ 30」試算日薪，乘以到期未休天數，僅供**預估參考**，正式結算建議再核實。
3. **請假天數計算**：以「日曆日」頭尾皆計入，未扣除例假日、國定假日。
4. **員工身份驗證**：用「姓名 + 身份證字號」比對查詢，**不是正式帳密登入**，安全性較低，適合內部信任環境使用。因為身份證字號並非機密（同事間可能互相知道），此驗證方式主要是防止「隨便打開網址就能亂查」，並非嚴謹的資安機制。
5. **隱私提醒**：「員工資料」分頁包含身份證字號、地址、緊急聯絡人電話等敏感個資，且身份證字號現在也會出現在「請假紀錄」分頁與員工登入查詢中。網頁本身不會把地址、緊急聯絡人等欄位顯示給員工或透過查詢外流，但**試算表本身的 Google 共用權限**建議只開放給 HR / 管理層，避免有人直接打開整份 Sheet 看到所有人的個資。
